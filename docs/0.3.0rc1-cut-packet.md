## Gate 0: Accept Physical-Disc QC

Issue #202 must contain an explicit accepted result for the immutable Beta 2
build. The report must identify the Mac, macOS version, optical drive, disc,
Beta 2 identity, and relevant logs, then cover:

- drive discovery and MakeMKV inspection;
- selected title, audio, subtitles, profile, and output mode;
- cancellation followed by safe eject or retry when practical;
- any recovery card that appeared and the resulting fresh job;
- one complete direct-disc conversion; and
- final playback and beginning, middle, and end seeking.

Do not infer a pass from an incomplete report. Route failures before release:

| Observation | RC decision |
| --- | --- |
| Drive is not discovered or does not refresh after insertion | Block; reproduce and repair the source workflow. |
| Inspection, title selection, or supported-source conversion fails because of app behavior | Block; preserve logs and open or update the owning issue. |
| The drive drops during heavy seeking | Retry once with a powered hub; block if app behavior remains implicated. |
| Stop leaves processing alive or prevents safe eject/retry | Block; treat cancellation cleanup as a release defect. |
| A recovery action mutates visible defaults, reuses a stale job, or cannot cancel cleanly | Block; repair recovery semantics. |
| Output does not play, seek, or preserve the expected stereo result | Block; preserve the output hash and playback evidence. |
| Conversion passes but required evidence is missing | Wait for the missing evidence; do not convert uncertainty into approval. |

If the run exercises the multithread MVC splitter, copy the unrelated-title and
product-UI evidence to issue #135 as well.

## Phase 1: Prepare Release Metadata

Begin only after Gate 0 is accepted. Create a focused branch from the latest
protected `main`, confirm no release workflow is active, and require a clean
worktree before running the preparation helper:

```sh
test -z "$(git status --porcelain=v1 --untracked-files=all)"
uv sync --locked --all-groups --python 3.12
uv run python scripts/release.py prepare \
  --version 0.3.0rc1 \
  --build 147
```

The command updates the first three release-owned files. The same release PR
must update the three intentional repository identity locks, for a six-file
boundary:

- `pyproject.toml`: project version `0.3.0rc1` and `CFBundleVersion` `147`;
- `uv.lock`: local package version `0.3.0rc1`, with no dependency-pin churn;
- `macos/project.yml`: `MARKETING_VERSION` `0.3.0rc1` and
  `CURRENT_PROJECT_VERSION` `147` for `BluRayToVisionPro`;
- `tests/test_release.py`: rename the repository identity test for the release
  candidate and expect RC1/build `147`, channel `rc`, prerelease `true`, Latest
  `false`, and PyPI `false`;
- `tests/test_native_app.py`: production app identity expects short version
  `0.3.0rc1` and build `147`; and
- `tests/test_sparkle_packaging.py`: repository public-key metadata expects
  build `147`.

Stop if any other file changes or if the product name, bundle identifier,
`SUFeedURL`, update policy, deployment target, workflow, or release environment
changes. Validate the prepared identity:

```sh
uv run python scripts/release.py validate
uv run python scripts/release.py metadata
git diff --check
git diff -- \
  pyproject.toml \
  uv.lock \
  macos/project.yml \
  tests/test_release.py \
  tests/test_native_app.py \
  tests/test_sparkle_packaging.py
```

Verify the complete worktree boundary rather than relying on the scoped diff:

```sh
EXPECTED_PATHS=$(printf '%s\n' \
  macos/project.yml \
  pyproject.toml \
  tests/test_native_app.py \
  tests/test_release.py \
  tests/test_sparkle_packaging.py \
  uv.lock | LC_ALL=C sort)
ACTUAL_PATHS=$(git status --porcelain=v1 --untracked-files=all \
  | cut -c4- \
  | LC_ALL=C sort)
if [ "$ACTUAL_PATHS" != "$EXPECTED_PATHS" ]; then
  printf 'Unexpected release PR paths:\n%s\n' "$ACTUAL_PATHS" >&2
  exit 1
fi
```

The metadata output must be equivalent to:

```json
{"build_version":"147","channel":"rc","make_latest":false,"package_version":"0.3.0rc1","prerelease":true,"publish_pypi":false,"release_name":"v0.3.0rc1","release_tag":"v0.3.0rc1"}
```

Run the focused release tests, then the broad practical local gates:

```sh
uv run python -m unittest \
  tests.test_release \
  tests.test_sparkle_workflows \
  tests.test_sparkle_packaging \
  tests.test_sparkle_appcast
```

Required CI additionally builds the direct SSIF probe, exercises a CLI-only
installation, runs Briefcase create/build smoke, and verifies bundled tools and
deployment targets. Open a release-identity-only pull request that references
#202. Merge it only after the required CI and CodeQL checks pass and review

## Phase 3: Dispatch and Approve

Dispatch **Release from protected main** on `main` through the configured
automation identity `shiny-code-bot`; do not dispatch or rerun it as `cbusillo`,
because the guarded approver rejects self-review. Paste the Proposed Release
Summary into the optional `release_notes` input. Immediately record the
resulting run ID, run attempt, actor, triggering actor, and full head SHA.
Verify both actors are `shiny-code-bot` and the SHA equals `MAIN_SHA` before
continuing.

When operating from Every Code, route the dispatch through the GitHub skill's
automation-auth wrapper rather than raw active-user `gh` authentication:

```sh
RELEASE_NOTES=$(cat <<'EOF'

skills_home="${CODE_HOME:-${CODEX_HOME:-$HOME/.code}}/skills"
"$skills_home/github/scripts/gh-with-env-token" workflow run briefcase.yml \
  --repo cbusillo/BD_to_AVP \
  --ref main \
  --field release_notes="$RELEASE_NOTES"
```

Use the Actions run URL or a recent-run lookup to obtain `RUN_ID`, then inspect
the run identity before invoking the watcher. Do not trust a merely latest run
without confirming the actor, triggering actor, workflow ID, and exact SHA.

Monitor only with the repository helper:

```sh
uv run python -m scripts.github_release_run watch \
  --run-id "$RUN_ID" \
  --workflow "Release from protected main" \
  --head-sha "$MAIN_SHA"
```

Exit code `20` means the expected `macos-signing` approval is waiting. Record
the emitted approval fingerprint and obtain explicit maintainer authorization
in the active conversation. Then approve the exact run:

```sh
uv run python -m scripts.github_release_run approve \
  --run-id "$RUN_ID" \
  --workflow "Release from protected main" \
  --head-sha "$MAIN_SHA" \
  --confirm-sha "$MAIN_SHA" \
  --approval-fingerprint "$APPROVAL_FINGERPRINT" \
  --comment "Approved after explicit release authorization for $MAIN_SHA."
```

Run the watcher again after approval. Do not approve if the workflow ID/path,
run attempt, actor, reviewer, environment, fingerprint, or SHA is unexpected.
Exit code `21` is a safety failure: stop or cancel rather than retrying blindly.

## Failure and Recovery

| Failure boundary | Required response |
| --- | --- |
| Before metadata merge | Fix the branch or wait; nothing has been released. |
| After metadata merge, before dispatch | Keep the committed RC identity; revalidate the final `main` SHA before dispatch. |
| `main` moves after dispatch | Stop or cancel. Do not approve, delete drafts, or redispatch until the exact run/draft state is reviewed. |
| Before publication | Preserve the draft and logs. Rerun failed jobs or redispatch only when the same commit and byte-identical draft are proven resumable. |
| After publication, RC defect found | Never replace RC1 assets or appcast snapshot. Prepare `0.3.0rc2` with build `148` after fixing and reviewing the defect. |
| Pages deployment fails after publication | Rerun the failed job or deploy the same published tag through **Manage Sparkle Pages**. |
| Live feed is unsafe | Use the reviewed **Manage Sparkle Pages** disable operation, then restore a verified last-good published snapshot when safe. |

Do not delete or mutate an immutable published release, manually edit the
appcast, upload replacement assets, remove the approval gate, or publish the RC
to PyPI as a recovery shortcut.
